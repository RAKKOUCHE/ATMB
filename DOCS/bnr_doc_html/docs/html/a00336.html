void Mei::Bnr::Bnr::Cancel ( )

Attempts to cancel the current asynchronous operation.

This command has no effect if the asynchrounous command is waiting for a user action after a CashAvailable event has been received. Use Bnr::CancelWaitingCashTaken() in this case.

If it can do so, an Bnr::OperationCompleted event is sent with the result field containing BnrXfsErrorCode::Canceled to indicate that the operation was canceled.

If this method is called after an operation has been completed but before the Bnr::OperationCompleted event has been returned to the caller, then no operation will take place and no error will be returned.

See also
Bnr::CancelWaitingCashTaken().
void Mei::Bnr::Bnr::CancelWaitingCashTaken ( )

Attempts to cancel the current asynchronous operation while the BNR is waiting for the user to remove cash at bezel.

If it can do so, an Bnr::OperationCompleted event is sent with the result field containing Canceled to indicate that the operation was canceled.

If this method is called after an operation has been completed but before the Bnr::OperationCompleted event has been returned to the caller, then no operation will take place and no error will be returned.

Firmware Compatibility
This function requires a BNR FW v1.5.0 or newer. With older FW versions, a NotSupported exception will be raised.
See also
Bnr::Cancel().
int Mei::Bnr::Bnr::CashIn ( )

This command will cause inserted notes to be moved to the escrow.

Notes will be held on the escrow until the current cashIn Transaction is either canceled by Bnr::CashInRollback() or confirmed by Bnr::CashInEnd() commands.

The notes may pass through the banknote reader for identification. Failure to identify notes does not mean that the command has failed - even if the banknote reader rejects some or all of the notes, the command may return Successful. In this case, an InputRefused intermediate event will be sent to listeners.

Note
When called without parameters, the cashIn method will cause the BNR to accept one banknote. After Bnr::OperationCompleted event, the host will calculate the amount that is still due by customer and call more cashIn if needed.
Returns
The identification Id if successful (strictly positive).
Exception:
  • Illegal - The BNR is in a state where it cannot execute the operation, e.g. another asynchronous command is already running, or a dispense is active (after a Bnr::Dispense() and before a Bnr::Present() call).
  • NoCashInStarted - A successfull Bnr::CashInStart() call was not done before calling Bnr::CashIn().
Other error code might be used in the future without notice. See BnrXfsErrorCode, BnrApiErrorCode, BnrUsbErrorCode for the full error code description.
StatusEvent event:
  • CashAvailable - Sent when a rejected bill is back at inlet, presented to the user for removal.
  • CashTaken - Sent when a bill has been taken by the user.
  • CashUnitChanged - Sent when a bill has been accepted by the BNR and escrowed in cash unit.
IntermediateEvent event:
The field reason can take the following values :
  • InputRefused - Sent if the note has not been identified and validated by the BNR. Data field contains BNR internal reason code.
  • SubCashin - Sent each time a banknote has been recognized in the case where amount parameter is not NULL.
OperationCompleted event:
The field result can take the following values :
  • Successful - The command has been executed successfully.
  • Canceled - The command has been interrupted because:
    • Bnr::Cancel() has been called during bill move.
    • The Cash Modules Lock or a cover has been opened.
    • Communication with the Spine module has been lost.
    • Communication with the Smart Cashbox module has been lost.
  • CashDeviceError - A failure occured during bills move, due to an unexpected error or a bill jam.
  • CashUnitError - Escrow full, the number of inserted bills equals the maximum escrow capacity.
  • Illegal - The BNR is in a state where it cannot execute the operation.
  • UserError - A person has stopped the device operation, i.e. manupulating the outlet flap or inserting a bill at the inlet when it is not expected.
Deprecated:
  • HardwareError - One of the electro-mechanical elements encountered an error during bill move - No more used since FW v1.3.0, use CashDeviceError instead.
If result field is Illegal, Data field is NULL. Otherwise, it is an XfsCashOrder Object with the following properties:
  • If all accepted bills are of the same currency, then XfsCashOrder.Currency holds the information of that currency; otherwise it is empty (meaning that exponent = 0, currencyCode consists of NULL-characters).
  • If all accepted bills are of the same currency, then XfsDenomination.Amount holds the sum of all the bill values; otherwise it is 0.
  • XfsDenomination.Cashbox is always 0.
  • XfsDenomination.Items holds the number of stored bills for each LCU involved.
This event is fired at the end of the operation.
See also
XfsCashOrder.

int Mei::Bnr::Bnr::Denominate ( int  amount,
String^  currencyCode,
ChangeAlgorithm  algorithm 
)

Asynchronous command to determine if the amount requested by value, is available for dispense.

Using ChangeAlgorithm, three methods are possible:

  • MinBills: The BNR chooses the banknotes to be distributed in order to obtain the total amount using the minimum number of banknotes.
  • OptimumChange: The BNR chooses the banknotes to be distributed in order to obtain the total amount in a way that slows down cashbox filling. As long as the low denomination Recyclers are not near to full, change is determined like with the MinBills algorithm. But when a Recycler becomes nearly full (5/6 of Full threshold), this algorithm will try to put more of this denomination in the change so that the Recycler doesn't become full and this denomination doesn't start to be cashed.
    • MixDenom: The user chooses through a list of Logical Cash Units, the banknotes to be distributed by the BNR in order to obtain the total amount. The following parameters must be correctly set:
      • denominateRequest->denomination.size gives the size of the items array
      • for each item of denominateRequest->denomination.items from 0 to (denominateRequest->denomination.size - 1):
        • denominateRequest->denomination.items[item].unit contains the number of a LCU from where banknotes must be distributed.
        • denominateRequest->denomination.items[item].count gives the number of banknotes to distribute from the LCU.
Parameters
[in]amountAmout to be dispensed, expressed in MDUs
[in]currencyCodeCurrency used. See this page for a full list of the existing ISO currency codes: http://www.iso.org/iso/en/prods-services/popstds/currencycodeslist.htm
[in]algorithmSelect the method used to determine if the requested banknotes are available.
Returns
If function call is successful return the identification Id.
Exception :
  • Illegal - The BNR is in a state where it cannot execute the operation.
  • InvalidMixNumber - The mix number specified in the request is unknown, or is MinBills or OptimumChange, but the requested amount is 0.
  • NotSupported - The unit is a BNA6.
Other error codes might be used in the future without notice.
OperationCompleted event:
The field result can take the following values :
  • BnrXfsErrorCode.Successful - The BNR has enough bills to satisfy the request.
  • NotDispensable - Not enough bills to satisfy the request.
  • TooManyBills - The number of bills needed to satisfy the request would exceed the BNR's limitation.
  • NoBills - The bill list passed as parameter was empty.
  • ParameterInvalid - At least one unitId in the list of specified bills is unknown.
  • Illegal - The BNR is in a state where it cannot execute the operation.
If the field result is ParameterInvalid or Illegal, the field data is NULL.
If the field result is NoBills, the field data is a XfsCashOrder Object with the fields XfsDenomination.Amount, XfsDenomination.Cashbox is 0 and XfsDenomination.Items is empty.
Otherwise, it is a XfsCashOrder Object with the following properties:
  • If all requested bills are of the same currency, then XfsCashOrder.Currency holds the information of that currency; otherwise it is empty (meaning that Exponent = 0, CurrencyCode consists of NULL-characters).
  • If not all requested bills are of the same currency, then the field XfsCDenomination.Amount is 0, otherwise it holds the sum of all requested bills (if result is Successful) or the amount which could be dispensed (if result is NotDispensable or TooManyBills).
  • If result is Successful or not all requested bills are of the same currency, XfsDenomination.Cashbox is 0; otherwise (result is NotDispensable or TooManyBills) it is equal to the amount requested minus the amount dispensable.
  • XfsDenomination.Items holds the number of stored bills for each LCU involved.
See also
Bnr::Denominate(array<XfsDenominationItem^>^ items), Bnr::Denominate(List<XfsDenominationItem^>^ items).

int Mei::Bnr::Bnr::Dispense ( int  amount,
String^  currencyCode,
ChangeAlgorithm  algorithm 
)

Dispenses the amount of money that is specified by the amount and currencyCode parameter.

The BNR will make a bundle of notes and wait for the Bnr::Present() command to give it to the customer.

Using ChangeAlgorithm, two methods are possible:

  • MinBills - The BNR chooses the banknotes to be distributed in order to obtain the total amount using the minimum number of banknotes.
  • OptimumChange - The BNR chooses the banknotes to be distributed in order to obtain the total amount in a way that slows down cashbox filling. As long as the low denomination Recyclers are not near to full, change is determined like with the MinBills algorithm. But when a Recycler becomes nearly full (5/6 of Full threshold), this algorithm will try to put more of this denomination in the change so that the Recycler doesn't become full and this denomination doesn't start to be cashed.
Parameters
[in]amountAmout to be dispensed, expressed in MDUs
[in]currencyCodeCurrency used. See this page for a full list of the existing ISO currency codes: http://www.iso.org/iso/en/prods-services/popstds/currencycodeslist.htm
[in]algorithmThe method used to select which banknote will be used.
Returns
If function call is successful return the identification Id.
Exception:
  • Illegal - The BNR is in a state where it cannot execute the operation, e.g. another asynchronous command is already running, or a successful Bnr::Dispense() call has already been done, etc...
  • CashInActive - A transaction started with Bnr::CashInStart() is already in progress. The transaction must be ended with Bnr::CashInEnd() before calling Bnr::Dispense().
  • InvalidMixNumber - The requested amount is 0.
Other error codes might be used in the future without notice.
OperationCompleted event:
The field result can take the following values:
  • Successful - The request is available and can be presented to the customer. The data field is a XfsCashOrder containing the amount and bill list that can be presented to the customer with a Bnr::Present() call.
  • CashUnitError - During bills move, a cash unit has less bills than expected (based on its bill count) at the beginning of the operation or the bundler is full. The data field is a XfsCashOrder containing the amount and bill list that may be presented to the customer with a Bnr::Present() call.
  • NotDispensable - Not enough bills to satisfy the request. No bill has been moved and the data field is a XfsCashOrder containing the maximum amount and bill list that could be used for this request.
  • TooManyBills - The number of bills needed to satisfy the request exceeds the BNR's limitation. No bill has been moved and the data field is a XfsCashOrder containing the list of bills that should be used for this request.
  • CashDeviceError - A failure occured during bills move, due to an unexpected error or a bill jam.
  • NoBills - The bill list passed as parameter was empty.
  • ParameterInvalid - At least one unitId in the list of specified bills is unknown.
  • Canceled - The command has been interrupted because:
    • The Cash Modules Lock or a cover has been opened.
    • Communication with the Spine module has been lost.
    • Communication with the Smart Cashbox module has been lost.
  • Illegal - The BNR is in a state where it cannot execute the operation.
  • UserError - A person has stopped the device operation, i.e. manupulating the outlet flap or inserting a bill at the inlet when it is not expected.
Deprecated:
  • HardwareError - One of the electro-mechanical elements encountered an error during bill move - No more used since FW v1.3.0, use CashDeviceError instead.
If result field is ParameterInvalid or Illegal, data field is NULL.
If result field is NoBills, data field is an XfsCashOrder Object with the fields XfsDenomination.Amount, XfsDenomination.Cashbox and XfsDenomination.Items is empty.
If result field is Successful, NoBills, NotDispensable or TooManyBills, data field is a XfsCashOrder Object with the following properties:
  • If all requested bills are of the same currency, then XfsCashOrder.Currency holds the information of that currency; otherwise it is empty (meaning that Exponent = 0, CurrencyCode consists of NULL-characters).
  • If not all requested bills are of the same currency, then the field XfsCashOrder.Denomination.Amount is 0, otherwise it holds the sum of all requested bills (if result is Successful) or the amount which could be dispensed (if result is NotDispensable or TooManyBills).
  • If result is Successful or not all requested bills are of the same currency, XfsCashOrder.Denomination.Cashbox is 0; otherwise (result is NotDispensable or TooManyBills) it is equal to the amount requested minus the amount dispensable.
  • XfsCashOrder.denomination.size holds number of elements in the items-array.
  • XfsCashOrder.denomination.items holds the number of stored bills for each LCU involved.
If result field is CashUnitError, CashDeviceError, or Canceled, data field is a XfsCashOrder Object with the following properties:
  • If all bills that made it into the Bundler are of the same currency, then XfsCashOrder.Currency holds the information of that currency; otherwise it is empty (meaning that Exponent = 0, CurrencyCode consists of NULL-characters).
  • If all bills that made it into the Bundler are of the same currency, then the field XfsDenomination.Amount in the XfsCashOrder, holds the sum of those bills, otherwise it is 0.
  • If all bills that did not make it into the Bundler are of the same currency, XfsDenomination.Cashbox holds the sum of those bills, otherwise it is 0 in the XfsCashOrder.
  • XfsDenomination.Items holds the number of stored bills for each LCU involved in the XfsCashOrder.

int Mei::Bnr::Bnr::Dispense ( array< XfsDenominationItem^ >^  items)

Dispenses the bill list given in parameter.

The application chooses through a list of Logical Cash Units the banknotes to be distributed by the BNR in order to obtain the total amount. For each XfsDenominationItem the two fields must be correctely set:

  • XfsDenominationItem::Unit contains the number of a LCU from where banknotes must be distributed.
  • XfsDenominationItem::Count gives the number of banknotes to distribute from the LCU.

The BNR will make a bundle of notes and wait for the Bnr::Present() command to give it to the customer.

Parameters
[in]itemsList of bills to be dispensed.
Returns
If function call is successful return the identification Id.
Exception:
  • Illegal - Another asynchronous command is already running, or asuccessful Bnr::Dispense() call has already been done.
  • CashInActive - A transaction started with Bnr::CashInStart() is already in progress. The transaction must be ended with Bnr::CashInEnd() before calling Bnr::Dispense().
Other error codes might be used in the future without notice.
OperationCompleted event :
Result field can take the following values :
  • Successful - The request is available and can be presented to the customer. The data field is a XfsCashOrder containing the amount and bill list that can be presented to the customer with a Bnr::Present() call.
  • CashUnitError - During bills move, a cash unit has less bills than expected (based on its bill count) at the beginning of the operation or the bundler is full. The data field is a XfsCashOrder containing the amount and bill list that may be presented to the customer with a Bnr::Present() call.
  • NotDispensable - Not enough bills to satisfy the request. No bill has been moved and the data field is a XfsCashOrder containing the maximum amount and bill list that could be used for this request.
  • TooManyBills - The number of bills needed to satisfy the request exceeds the BNR's limitation. No bill has been moved and the data field is a XfsCashOrder containing the list of bills that should be used for this request.
  • CashDeviceError - A failure occured during bills move, due to an unexpected error or a bill jam.

    • Communication with the Smart Cashbox module has been lost.
  • Illegal - The BNR is in a state where it cannot execute the operation.

int Mei::Bnr::Bnr::Eject ( )

This command allows the application to force cash that has been presented to be ejected from the bezel.

  • If this command is called during a CashIn transaction, the application can then continue this transaction, using Bnr::CashIn(), Bnr::CashInRollback() or Bnr::CashInEnd();
  • If this command is called during a Dispense transaction, then the transaction is finished.

This method may only be called after bills have been presented at the outlet following a Bnr::CashIn() (bill has been refused), Bnr::Dispense() (if autoPresent mode is active), Bnr::CashInRollback() or Bnr::Present() method call, and before the bills have been taken by the user.

Note
An asynchronous method must not be called before the preceding one is terminated (i.e. OperationComplete event has been received); typically before calling Bnr::Eject(), the preceding command must be terminated by calling Bnr::Cancel().
Returns
If function call is successful, return is strictly positive and represents the identificationId.
Exception :

int Mei::Bnr::Bnr::Retract ( )

This command allows the application to force cash that has been presented to be retracted.

Retracted bills will be moved to the intermediate stacker area and accounted in the Bundler LCU. The application can then present bills to the user, using Bnr::CashInRollback() or Bnr::Present() (depending of the kind od the transaction) or clear the intermediate stacker area using the Bnr::Reject() method.

This method may only be called after bills have been presented at the outlet following a Bnr::Dispense() (if autoPresent mode is active), Bnr::CashInRollback() or Bnr::Present() method call, and before the bills have been taken by the user.

int Mei::Bnr::Bnr::SelfTest ( )

Allows BNR to perform self tests in order to stay operational.

This command should be called on a regular basis only if Capabilities.SelfTestMode is set to SelfTestMode::Slave (Auto by default). The self tests include:

  • static tests
  • automatic recovery movements to clear an error condition
  • repositionning of internal devices that lost idle status. Example: Bundler drum, Piston Stacking Mech.
  • Refloat of Recyclers from Loader. All these operations may include movements.

This command is canceled by the Bnr::Cancel() command and is terminated by a Bnr::OperationCompleted event. In this case, any movement in progress is terminated (ex: bill transfer from Loader to Recycler) and the BNR stops any other self test.

Recommendation:
As soon the BNR is not used in a transaction, Bnr::SelfTest() should be called and not canceled before the next transaction.
If this cannot be applied, Bnr::SelfTest() should be called typically every 20 seconds (but at least once per minute) and can be canceled one second after the call. This will insure a BNR self test can be done.
Returns
The identificationId.
Events:
  • OperationCompleted
Exceptions
BnrXfsExceptionThrow BnrXfsErrorCode::Illegal if:
  • the BNR is in exchange state
  • the BNR is in cash in state
  • the BNR is in dispense state
  • the property Capabilities.SelfTestMode in defined to SelfTestMode::Auto
